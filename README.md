# 🛡️ FraudShield — Retail Transaction Fraud Detection

A machine-learning app that assesses a retail card transaction and estimates the
probability that it is fraudulent. Enter a transaction's details — amount, recent
activity, device, location, and risk flags — and get an instant verdict.

---

## What this project does

Given a single transaction, the model predicts whether it is **fraud** (a card
being used by someone other than the genuine owner) or **legitimate** (a normal,
safe purchase). It was built end-to-end: data cleaning, exploratory analysis,
feature engineering, model comparison, and a deployed interactive app.

---

## The data

- Source: Kaggle — `retail_fraud_detection_100k.csv`
- Size: **100,000 transactions × 21 columns**, one year of activity
- Target: `fraud_flag` (1 = fraud, 0 = legitimate)

### Key findings during cleaning
1. **No nulls, no duplicates, clean categories** — no imputation needed (verified, not assumed).
2. **Fraud rate is 47.5%** — artificially balanced. Real card fraud is under 1%, so
   any accuracy here will not transfer directly to production.
3. **`fraud_risk` is target leakage** — "Low" is always legitimate, "High" is always
   fraud. It is a coarsened copy of the answer, so it is **excluded from model features**.
4. **`unusual_location_flag` was identical to `is_international`** — duplicate column dropped.
5. **Outliers kept on purpose** — large/unusual amounts are the fraud signal, not noise.
6. **`account_age_days` carried no signal** and was dropped from the model.
7. **Timestamp is non-informative** (constant hour) — no time features engineered.

## Feature engineering

Three features were built and validated against the data:

| Feature | What it captures | Why it matters |
|---|---|---|
| `amount_vs_personal_avg` | transaction ÷ customer's 7-day average | Flags spending that's abnormal *for that person* (fraud ≈ 2× their norm) |
| `risk_flag_count` | sum of the 6 risk flags | Stacked warnings are far stronger than any single one — **top feature in the model** |
| `amount_x_frequency` | amount × transactions in 24h | Captures the "large + fast" account-takeover pattern |

Two candidate features (`failed_transaction_ratio`, `customer_transaction_count`)
were tested and **rejected** for ~zero correlation — documenting failed experiments
is part of the rigour.

## Models & evaluation

Two models were compared:
- **Logistic Regression** — interpretable linear baseline.
- **Random Forest** — captures interactions, robust to outliers, gives feature importance.

**Metrics:** accuracy is *not* the headline because fraud's two errors cost very
differently (a missed fraud = financial loss; a blocked legit customer = lost sale).
Reported instead: **precision, recall, F1, ROC-AUC, PR-AUC, and the confusion matrix.**

**Result & the key insight:** both models score ~100% on this dataset. That is a
**red flag, not a trophy** — the fraud label was generated by a deterministic rule
from the features, so any capable model recovers it exactly. A 15% label-noise
stress test (to mimic a realistic, non-separable world) brought both models to
~0.84 F1, essentially tied, so **Logistic Regression is preferred** — equal
performance, more explainable.

## Try it yourself (test scenarios)

The prediction is driven mainly by **how many risk flags are set** (the strongest
signal in the data). Use these inputs in the live app to verify it works:

| Scenario | Inputs | Expected result |
|---|---|---|
| ✅ Safe | Amount £120, freq 5, no flags ticked | ~0% fraud |
| ✅ Safe | Even a large amount (£4000) with **no** flags | still low (~6%) |
| ⚖️ Borderline | Any **2** risk flags ticked | ~50% — a coin flip |
| ⚠️ Fraud | Any **3** risk flags ticked | ~99% fraud |
| ⚠️ Fraud | **4+** flags ticked | 100% fraud |

Note: predictions snap toward 0% or 100% rather than fuzzy values — that is the
fingerprint of this deterministically separable dataset, and is expected.

## What would break in production

- **The accuracy is unrealistic.** Real fraud is < 1% and never perfectly separable;
  precision would drop sharply on a live stream.
- **The model leans heavily on the risk flags.** In production those flags must be
  generated upstream and in real time; if they're delayed or missing, the model is weak.
- **No concept drift handling.** Fraudsters adapt; this model would need regular retraining.
- **Threshold matters.** The 50% cut-off should be tuned to the business's cost of a
  missed fraud vs. a blocked customer, not left at the default.
